# Bug 檢查報告（僅分析，未直接修改程式）

## 1) `save_setting` 在「新 key 不存在」時不會 append，會直接失敗
- **位置**：`app.py` 第 756–767 行。
- **問題**：`ws.find(key)` 在 gspread 找不到時通常會丟出 `CellNotFound` 例外，不會回傳 `None`。目前流程把 `find` 與 `append_row` 放在同一個 `try`，因此找不到 key 時會直接進 `except` 並 `return False`，導致無法新增新設定。
- **風險**：後台新增設定項（例如新參數）可能永遠存不進去。
- **建議修正方向**：
  1. 專門捕捉 `gspread.exceptions.CellNotFound`，找不到時改走 `append_row`。
  2. 或先拉整張設定表做 key 比對，再決定 update/append。

## 2) `update_appeal_status` 以 `record_id + 待處理` 找列，可能更新錯誤申訴
- **位置**：`app.py` 第 818–841 行。
- **問題**：函式簽名有 `idx` 但未使用。實際是透過 `對應紀錄ID + 處理狀態=待處理` 找第一筆，若同一紀錄有多筆申訴，會命中錯誤列。
- **風險**：審核結果可能套用到不是使用者點選的那一筆，造成資料一致性問題。
- **建議修正方向**：
  1. 優先以唯一鍵 `申訴ID` 定位。
  2. 若前端已有 row index，應直接用 `idx` 映射到 Google Sheet 列號。

## 3) 背景任務在「主表已寫入、時數寫入失敗」時會重試，可能重複寫主表
- **位置**：`app.py` 第 536–546 行、568–605 行。
- **問題**：`process_task` 對 `main_entry/volunteer_report` 先 `_append_main_entry_row(entry)`，再寫入服務時數。若後半段失敗，整個 task 會標記 RETRY。重試後主表會再 append 一次，形成重複紀錄。
- **風險**：主表重複列、分數重複計算、後續申訴/統計對錯資料。
- **建議修正方向**：
  1. 以 `紀錄ID` 做主表去重（寫入前先查是否存在）。
  2. 或拆成兩個可獨立重試且具 idempotency 的 task（主表寫入 task / 時數寫入 task）。
  3. 任務狀態紀錄「主表已完成」旗標，重試時跳過已完成步驟。

## 4) `save_entry` 圖片儲存失敗時仍回傳成功並入隊
- **位置**：`app.py` 第 926–953 行。
- **問題**：單張檔案失敗只 `print`，函式不會中止；即使全部檔案都失敗，仍會 `enqueue_task(...)` 並回傳 `True`。
- **風險**：前端顯示成功，但實際無照片，造成稽核證據缺失。
- **建議修正方向**：
  1. 新增 `failed_files` 計數；若有失敗且該流程要求照片，直接回傳錯誤。
  2. 至少在全部失敗時阻擋入隊，並 `st.error` 提示重傳。
